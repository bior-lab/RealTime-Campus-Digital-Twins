# Urban Digital Twins Data Reuse Assessment

## Source

Repository:

```text
https://github.com/bior-lab/Urban-Digital-Twins
```

The repository describes itself as a static Mapbox GL JS site for Singapore building energy, 500 m district energy, and WRF-derived microclimate layers.

## Confirmed Repository Content

### License

The repository uses the MIT license.

### Main Site Function

The main site includes:

- Singapore building energy layers;
- 500 m district energy layers;
- WRF-derived microclimate layers;
- hourly weather playback through JSON time-series files;
- Mapbox GL JS frontend.

### Mapbox Tileset Configuration

The current `src/config.js` points to Mapbox tilesets under the `nus-bior` Mapbox account:

```text
buildings:        mapbox://nus-bior.7xmymq8g
grid:             mapbox://nus-bior.0siwr5r9
weather:          mapbox://nus-bior.7x1jhs36
buildingOverview: mapbox://nus-bior.coxvspif
```

The current campus EMIS prototype uses these tilesets as optional Mapbox GL JS vector sources:

| Frontend source | Tileset URL | Source layer | Primary use |
| --- | --- | --- | --- |
| `udt-buildings` | `mapbox://nus-bior.7xmymq8g` | `07_buildings_energy_weekly_re-1cntpw` | Singapore building digital twin context |
| `udt-grid` | `mapbox://nus-bior.0siwr5r9` | `06_grid_energy_weekly-dipyjj` | 500 m energy grid context |
| `udt-weather` | `mapbox://nus-bior.7x1jhs36` | `03_weather_500m-bt47nt` | 500 m microclimate context |
| `udt-building-overview` | `mapbox://nus-bior.coxvspif` | `04_building_overview_500m-cu3xd0` | 500 m building stock summary |

For each vector tileset, Mapbox GL JS needs both the source URL and the `source-layer` name. The source-layer name is generated by Mapbox during upload and is not always the same as the short tileset ID.

## Reuse Decision

### Can Reuse

Use this repository for:

- Mapbox frontend structure;
- layer switching logic;
- Singapore-scale context layer;
- 500 m grid workflow;
- WRF weather time-series structure;
- Mapbox Studio upload workflow;
- metadata and legend design pattern.

### Cannot Directly Reuse

Do not treat it as a complete NUS campus building model because:

- it is Singapore-scale, not campus-scale;
- the building layer is not linked to NUS EMIS meter IDs;
- raw production GeoJSON is not fully stored in GitHub;
- existing Mapbox tilesets are tied to the `nus-bior` Mapbox account;
- the data is not a BIM or asset-level building model;
- real-time InfluxDB/FastAPI energy data is not part of the repository.

## Recommended Role in NUS Campus EMIS

Use the Singapore-scale layers as context:

- city boundary and urban surroundings;
- 500 m grid context;
- microclimate context;
- reference building morphology.

In the current prototype, the UDT building twin is visible by default. The grid, weather, and building-overview layers are available as optional toggles. The campus EMIS layer remains the primary analytical layer.

Create a separate NUS campus layer for:

- campus boundary;
- NUS building footprints;
- building ID mapping;
- EMIS meter groups;
- current energy status;
- forecast and anomaly data;
- price and carbon indicators.

## Next Data Steps

### Step 1. Confirm Mapbox Tileset Access

Check whether the new `nus-campus-emis-dev` Mapbox token can read the existing `nus-bior` tilesets.

If not, either:

- upload required layers to the current Mapbox account; or
- use GeoJSON/API layers from our own server.

### Step 2. Extract NUS Campus Buildings

Use one of these approaches:

1. Extract NUS buildings from the Singapore building layer by spatial boundary.
2. Use NUS GIS/building footprint data if available.
3. Create a campus GeoJSON manually for the first prototype.
4. Combine building footprint data with EMIS building metadata.

### Step 3. Build ID Mapping

Create a mapping table:

```text
map_building_id
nus_building_name
emis_building_id
meter_group_id
zone
building_type
area_m2
public_visibility
```

### Step 4. Connect EMIS API

Use the FastAPI JSON pipeline:

```text
InfluxDB -> FastAPI -> cloud API -> Mapbox frontend
```

The campus map should load static geometry once and refresh live EMIS indicators on a schedule.

### Step 5. Decide Deployment Path

For GitHub and cloud deployment:

1. Push the website code to the lab GitHub repository.
2. Keep public tokens restricted by domain.
3. Keep API secrets out of GitHub.
4. Serve the frontend through the cloud server or static hosting.
5. Configure FastAPI CORS for the final website domain.
